## Identity

**Category:** tooling / developer tools
**Mood:** minimal, technical
**Character:** precise, quiet, tactile, and engineering-led. The components provide the personality; the documentation interface stays neutral.

## Palette — beUI Current

The palette is intentionally near-monochrome. Cyan is the functional accent, violet is a secondary visual accent, and status colors remain semantic.

### Core seeds

| Role | Light | Dark |
|---|---|---|
| Background | `oklch(99% 0 0)` / `#FCFCFC` | `#151515` |
| Elevated surface | `oklch(97% 0 0)` / `#F5F5F5` | `#1C1C1C` |
| Foreground / primary | `oklch(15% 0 0)` / `#0B0B0B` | `oklch(96% 0 0)` / `#F2F2F2` |
| Cyan accent | `oklch(72% 0.18 195)` | `oklch(80% 0.18 195)` |
| Violet accent | `oklch(68% 0.22 295)` | `oklch(68% 0.22 295)` |

### Semantic tokens

The source of truth is `app/globals.css`. Components must consume semantic utilities such as `bg-background`, `bg-card`, `text-foreground`, `text-muted-foreground`, `border-border`, and `ring-accent` rather than repeating color values.

The full light and dark token sets are applied to `:root` and `.dark`. The pre-brand backup is stored at `app/globals.css.bak`.

### Contrast

| Pair | Light | Dark |
|---|---:|---:|
| Foreground / background | 19.12:1 | 16.44:1 |
| Muted foreground / background | AA | AA |
| Primary foreground / primary | 18.57:1 | 17.51:1 |

Borders are structural and intentionally subtle. Never rely on a border alone to communicate state. Interactive focus states must use the visible foreground or accent ring.

## Typography — Geist + Geist Mono

- **Display and body:** Geist
- **Code, commands, numbers:** Geist Mono

Both are wired through `next/font/google` in `app/layout.tsx` and exposed through `--font-sans`, `--font-display`, and `--font-mono` in `app/globals.css`.

### Type roles

| Role | Class | Use |
|---|---|---|
| Display | `text-5xl font-semibold tracking-tight` | Landing hero only |
| Page title | `text-3xl font-semibold tracking-tight` | Component and category titles |
| Section title | `text-xl font-semibold` | Major documentation sections |
| Subsection | `text-base font-medium` | Examples and grouped controls |
| Body | `text-sm` | Default interface copy |
| Reading | `text-base leading-7` | Longer explanations |
| Caption | `text-xs text-muted-foreground` | Metadata and supporting labels |
| Code | `font-mono tabular-nums` | Commands, source, and changing numbers |

## Layout and surfaces

- Documentation pages use generous horizontal space and restrained vertical rhythm.
- Main reading text should remain comfortably narrow even when previews and code use the full available width.
- Cards use `bg-card`, subtle borders, and no heavy shadows.
- Use rounded surfaces consistently. Pills are reserved for compact controls, badges, and active navigation.
- Prefer alignment, spacing, and typography over decorative containers.

## Gradients

Two optional gradient tokens are available as `bg-brand-bg` and `bg-brand-accent`.

### Background gradient

`--gradient-bg` is a nearly invisible neutral-to-cyan wash for large headers or empty hero areas.

### Accent gradient

`--gradient-accent` moves from the cyan accent to violet. Use it for small visual accents, feature headers, or decorative borders—not behind body text.

Use at most one gradient on a screen. Do not animate gradients and do not make gradients the default treatment for cards or buttons.

## Tone and voice

### Use

Write in a sparse, precise, active voice. Name the component, state what it does, and show the install command. Prefer concrete behavior over adjectives and lists over long paragraphs.

### Avoid

Avoid hype and effort language: “revolutionary,” “powerful,” “seamless,” “simply,” “just,” “unlock,” and “supercharge.” Avoid exclamation marks and vague claims about quality.

### Example

> A spring-pressed button with loading, success, and error states. Copy the source or install it from the registry.

## Usage rules

**Do:**

- Use semantic color and typography tokens.
- Keep the documentation chrome neutral so component motion remains the focal point.
- Use a 4 px spacing grid and consistent alignment across previews, code, and navigation.
- Test light and dark mode at mobile, tablet, and desktop widths.
- Use Geist Mono for install commands, code, and tabular numbers.
- Keep focus states visible and respect reduced motion.

**Don't:**

- Hardcode colors in component or documentation files when a token exists.
- Add decorative gradients, shadows, or glass treatments by default.
- Use `transition: all`.
- Mix multiple radius systems within one surface group.
- Duplicate implementation rules from `AGENTS.md`; that file remains authoritative for motion, accessibility, registry, and code conventions.

## Ownership

- `brand.md`: visual language, typography, layout character, and voice.
- `AGENTS.md`: repository workflow, implementation standards, motion behavior, accessibility, and registry contracts.
